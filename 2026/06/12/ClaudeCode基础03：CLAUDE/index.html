ClaudeCode基础03：CLAUDE.md

发表于2026-06-12|更新于2026-06-12|ClaudeCode基础
|总字数:1.7k|阅读时长:5分钟|浏览量:

VERSION：1.0.0

主要作用

CLAUDE.md 是每个新会话默认注入到上下文的文件，用于快速向 AI 交代项目背景。

加载机制

  • 主文件夹（~/.claude/CLAUDE.md）：适用于所有 Claude 会话
  • 项目根目录（./CLAUDE.md）或（.claude/CLAUDE.md）：提交到 git 以便与团队共享
  • 当前目录级：在当前正在编辑的文件夹下放置 CLAUDE.md，适用于针对特定模块的补充配置。
    越具体的目录优先级越高（即子目录 > 项目根目录 > 全局用户级），即越靠近正在编辑的代码的CLAUDE.md，优先级越高。

使用须知

  • CLAUDE.md 是一个特殊文件，Claude 在每次对话开始时读取。包括 Bash 命令、代码风格和工作流规则。这给 Claude 提供了它无法从代码中推断的永久上下文。
  • CLAUDE.md 文件没有强制的格式，但保持简短和易读。
  • CLAUDE.md 在每个会话中加载，所以只包括广泛适用的东西。对于仅有时相关的域知识或工作流，改用 skills。Claude 按需加载skill，不会使每次对话都加载。
  • 保持简洁。对于每一行，问自己：“删除这个会导致 Claude 犯错吗？” 如果不会，删除它。
  • 膨胀的 CLAUDE.md 文件会导致 Claude 忽略你的实际指令！

内容规范

一份优秀的 CLAUDE.md 通常涵盖以下四个层面：

  1. WHAT（是什么）：明确技术栈、框架版本、包管理器，防止 AI 给出过时的技术方案或放错代码位置。
  2. WHY（为什么）：简述项目的目录结构、各模块的功能等，帮助 AI 更好的了解业务。
  3. HOW（怎么做）：提供标准化的构建、测试、类型检查和启动命令（如 pnpm dev, npm run test），让 AI 知道如何验证改动是否正确。
  4. 边界与约束：明确告诉 AI “什么不能做”，以及一些遇到过的陷阱或坑，例如禁止修改某些底层目录、禁止引入未经讨论的新依赖等。

编写建议

  • 运行 /init 根据你的当前项目结构生成启动 CLAUDE.md 文件，然后随时间逐步完善。
  • 谨慎使用 AI生成CLAUDE.md，建议手动精雕细琢进行完善。
  • 避免层级间的矛盾规则：如果在不同层级的文件中写了相互矛盾的规则，AI 的行为会变得不确定甚至随机选择。
  • 定期精简与测试：像对待代码一样对待 CLAUDE.md。每删除一行都应自问：“删除这行会导致 AI 出错吗？”如果不会就删掉。可以通过观察 AI 行为的变化来持续测试和优化你的CLAUDE…md。
  • 少即是多：模型的指令遵循能力是有限的。建议将文件长度控制在 200-300 行以内，只保留对所有任务普遍适用的核心指令。
  • 不要把所有细节都塞进来。对于低频或仅特定场景适用的知识，应建立独立文档（如 agent_docs/ 目录），在 CLAUDE.md 中仅保留指针，指示 Claude 按需读取。
    |✅ 包括|❌ 排除|
    |—|—|
    |Claude 无法猜测的 Bash 命令|Claude 可以通过读取代码弄清楚的任何东西|
    |与默认值不同的代码风格规则|Claude 已经知道的标准语言约定|
    |测试指令和首选测试运行器|详细的 API 文档（改为链接到文档）|
    |存储库礼仪（分支命名、PR 约定）|经常变化的信息|
    |特定于你的项目的架构决策|长解释或教程|
    |开发者环境怪癖（必需的环境变量）|自明的实践，如"编写干净的代码"|
    |常见陷阱或非显而易见的行为|文件逐个描述代码库|
