需求场景

  • 如果一件事你每天做超过一次，就应该把它变成 skill
  • 当你不断将相同的说明、检查清单或多步骤程序粘贴到聊天时，创建一个 skill
  • 当 CLAUDE.md 的部分内容已经演变成程序时，创建一个 skill

skills 的位置

位置 路径 适用于
个人 ~/.claude/skills/<skill-name>/SKILL.md 你的所有项目
项目 .claude/skills/<skill-name>/SKILL.md 仅此项目
插件 <plugin>/skills/<skill-name>/SKILL.md 启用插件的位置

skills 更新后如何生效

  • Claude Code 监视 skill 目录的文件变更。
  • 在 ~/.claude/skills/、项目 .claude/skills/ 或 --add-dir 目录内的 .claude/skills/ 中添加、编辑或删除 skill 会在当前会话中生效，无需重新启动。
  • 创建在会话启动时不存在的顶级 skills 目录需要重新启动 Claude Code，以便可以监视新目录。

    对于[插件]中的 skill 文件夹，对 hooks/、.mcp.json、agents/ 和 output-styles/ 的更改需要 /reload-plugins 才能生效。

skills 的大致类别

对 Anthropic 内部的所有技能做了归类，发现它们可以清晰地划分为 9 个类别。
优秀的 skills 往往精准对应其中一类；而试图覆盖太多场景的技能会横跨多个类别，反而让智能体产生混淆。
这并非绝对标准的分类，但可以作为你搭建 skills 时查漏补缺的实用框架。

类别 说明 示例
库与 API 参考 内部库、CLI、SDK 的使用说明与避坑指南 billing-lib、platform-cli、events
产品验证 驱动运行中的产品完成功能验证 signup-driver、checkout、admin
数据与分析 封装 ID、字段名、常用查询模式 funnel-query、grafana、datadog
业务自动化 多工具工作流整合为单条命令 standup、tickets、weekly-recap
代码质量与评审 保障代码交付质量的方法论 adversarial、hypothesis、bughunt
CI/CD 与部署 安全可控的提交、推送与发布流程 babysit-pr、deploy、cherry-pick
基础设施运维 带安全门禁的清理与维护操作 orphans、deps、cost-investigation
脚手架与模板 符合框架规范的样板代码生成 new-app、migration、workflow
故障排查手册 从症状定位到调查再到报告输出的完整流程 oncall、correlator、queue-debug

库与 API 参考

这类技能用于说明如何正确使用某个库、CLI 工具或 SDK，既可以是内部私有库，也可以是 Claude Code 容易用错的通用库。这类技能通常会包含一个参考代码片段文件夹，以及一份 Claude 编写脚本时需要规避的坑点清单。
典型示例：

  • billing-lib —— 内部计费库：边界场景、常见陷阱等
  • internal-platform-cli —— 内部 CLI 封装的全部子命令，附带适用场景示例
  • sandbox-proxy —— 开发环境出口网关配置：可访问的主机列表、“连接被拒绝”报错排查方法、白名单新增流程

产品验证

这类技能定义了测试与验证代码功能的方法，通常会结合 Playwright、tmux 或其他外部工具完成验证。
在内部实践中，验证类技能对 Claude 输出质量的提升效果最可量化。投入一名工程师一周的时间打磨验证技能，通常是非常值得的。
可以采用的技巧包括：让 Claude 录下执行过程的视频，让你直观看到它测试了哪些内容；或者在每一步对系统状态做程序化断言。这些能力通常通过技能中内置的各类脚本实现。
典型示例：

  • signup-flow-driver —— 在无头浏览器中跑完「注册 → 邮箱验证 → 新手引导」全流程，每一步都内置状态断言钩子
  • checkout-verifier —— 使用 Stripe 测试卡驱动结账界面，验证发票最终进入正确状态
  • tmux-cli-driver —— 用于需要 TTY 终端的交互式 CLI 测试场景

数据获取与分析

这类技能对接你的数据与监控体系，可能包含带鉴权的数据获取库、特定仪表盘 ID，以及常用工作流说明或数据获取方式。
典型示例：

  • funnel-query —— 解答“需要关联哪些事件才能看到注册 → 激活 → 付费的转化漏斗”，并给出包含标准 user_id 的对应数据表
  • cohort-compare —— 对比两个用户群的留存或转化率，标记统计显著的差异，给出分群定义的跳转链接
  • grafana —— 数据源 UID、集群名称、问题 → 仪表盘对照表
  • datadog —— 字段参考（@request_id 与 trace_id 的区别）、服务列表、指标前缀命名规范

业务流程与团队自动化

这类技能把重复性工作流封装成单条命令。它们的指令逻辑通常很简单，但可能依赖其他技能或 MCP 工具。对于这类技能，把历史执行结果存入日志文件，有助于模型保持输出一致性，也能对历史执行结果做复盘。
典型示例：

  • standup-post —— 聚合工单系统、GitHub 活动记录与历史 Slack 消息 → 生成仅含增量内容的格式化站会日报
  • create-<ticket-system>-ticket —— 强制字段规范（合法枚举值、必填字段），并执行创建后流程（@审核人、Slack 中同步链接）
  • weekly-recap —— 合并的 PR + 关闭的工单 + 发布记录 → 格式化周报

代码脚手架与模板

这类技能为代码库中的特定功能生成框架样板代码。你可以把技能与可组合的脚本结合使用。当你的脚手架需求包含自然语言描述、无法纯代码覆盖时，这类技能尤其有用。
典型示例：

  • new-<framework>-workflow —— 生成带有自定义注解的新服务/工作流/处理器脚手架
  • new-migration —— 你的数据库迁移文件模板 + 常见坑点
  • create-app —— 预置好鉴权、日志、部署配置的内部新应用模板

代码质量与评审

这类技能在团队内强制执行代码质量标准，辅助代码评审。可以内置确定性脚本或工具来最大化可靠性。你可以将这类技能设为钩子自动执行，也可以集成到 GitHub Action 中。
典型示例：

  • adversarial-review —— 启动一个全新视角的子智能体做代码批判，落地修复方案，迭代直到剩余问题都属于细枝末节
  • code-style —— 强制执行代码风格，尤其是 Claude 默认容易出错的风格规范
  • testing-practices —— 关于如何写测试、测试哪些内容的指导说明

CI/CD 与部署

这类技能辅助你在代码库中拉取、提交和部署代码，可能会引用其他技能来收集数据。
典型示例：

  • babysit-pr —— 监控 PR 状态 → 重试不稳定的 CI → 解决合并冲突 → 开启自动合并
  • deploy-<service> —— 构建 → 冒烟测试 → 灰度放量并对比错误率 → 出现回归自动回滚
  • cherry-pick-prod —— 隔离工作区 → 遴选提交 → 冲突解决 → 带模板的 PR

故障排查手册

这类技能接收一个症状（比如 Slack 讨论串、告警、错误特征），引导跨工具排查，最终输出结构化报告。
典型示例：

  • <service>-debugging —— 为高流量服务整理「症状 → 工具 → 查询模式」的映射关系
  • oncall-runner —— 获取告警信息 → 排查常见诱因 → 格式化输出排查结论
  • log-correlator —— 输入一个请求 ID，拉取所有经手系统中匹配的日志

基础设施运维

这类技能执行日常维护与运维操作，其中部分是破坏性操作，适合加上安全护栏。它们能让工程师在关键操作中更容易遵循最佳实践。
典型示例：

  • <resource>-orphans —— 发现孤儿 Pod/卷 → 同步到 Slack → 缓冲观察期 → 用户确认 → 级联清理
  • dependency-management —— 团队的依赖审批工作流
  • cost-investigation —— 解答“为什么存储/出口费用暴涨”，附带具体的费用分类与查询模式
