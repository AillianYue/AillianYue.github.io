ClaudeCode基础02：使用技巧

发表于2026-06-11|更新于2026-06-26|VibeCodingClaudeCode
|总字数:3.7k|阅读时长:11分钟|浏览量:

VERSION：1.0.1

大多数使用技巧都基于一个前提：Claude 随着上下文窗口的填充，大模型性能会下降。

给 Claude 工作完成的标准

给 Claude 一个它可以运行的检查：测试、构建、屏幕截图进行比较。这是你观看的会话和你可以离开的会话之间的区别。

如果你没给验收标准，那么当工作看起来完成时，Claude 会停止。没有它可以运行的检查，"看起来完成"是唯一可用的信号。
如果你给了验收标准，那么就有了验证工作是否完成的循环：Claude 完成工作，运行检查，读取结果，并迭代直到检查通过。下面是３个示例：

  1. 提供验证标准
    1. 反面教材：实现一个验证电子邮件地址的函数
    2. 正面教材：编写一个 validateEmail 函数。示例测试用例：user@example.com 为真，invalid 为假，user@.com 为假。实现后运行测试
  2. 以视觉方式验证 UI 更改
    1. 反面教材：让界面看起来更好
    2. 正面教材：(粘贴屏幕截图)实现此设计。对结果进行屏幕截图并与原始设计进行比较。列出差异并修复它们
  3. 解决根本原因，而不是症状
    1. 反面教材：构建失败了
    2. 正面教材：构建失败，出现此错误：(粘贴错误)。修复它并验证构建成功。解决根本原因，不要抑制错误

让 Claude 显示工作完成的证据而不是只是嘴上说工作完成了。

给 Claude 具体的上下文

你的指令越精确，你需要的更正就越少。

Claude 可以推断意图，但它不能读心术。引用特定文件、提及约束，并指出示例模式。你的指令越精确，你需要的更正就越少。下面是４个示例：

  1. 限定任务范围。 指定哪个文件、什么场景和测试偏好
    1. 反面教材：为 foo.py 添加测试
    2. 正面教材：为 foo.py 编写测试，涵盖用户已注销的边界情况。避免 mock
  2. 指向来源。 指导 Claude 到可以回答问题的来源
    1. 反面教材：为什么 ExecutionFactory 有这样奇怪的 API ？
    2. 正面教材：为 foo.py 编写测试，涵盖用户已注销的边界情况。避免 mock
  3. 参考现有模式。 指向代码库中的模式
    1. 反面教材：添加日历小部件
    2. 正面教材：查看主页上现有小部件的实现方式以了解模式。HotDogWidget.php 是一个很好的例子。按照模式实现一个新的日历小部件，让用户选择月份并向前/向后分页以选择年份。从头开始构建，除了代码库中已使用的库外，不使用其他库。
  4. 描述症状。 提供症状、可能的位置以及"修复"的样子
    1. 反面教材：修复登录错误
    2. 正面教材：用户报告会话超时后登录失败。检查 src/auth/ 中的身份验证流程，特别是 token 刷新。编写一个失败的测试来重现问题，然后修复它

但是，当你在探索改正方向时，模糊的提示可能更有用。像"你会改进这个文件的什么？"，如果不是，那请在提示中提供具体的上下文

提供丰富的内容

使用 @ 引用文件、粘贴屏幕截图/图像或直接管道数据。

你可以通过多种方式向 Claude 提供丰富的数据：

  • 使用 @ 引用文件，而不是描述代码的位置。Claude 在响应前读取文件。
  • 直接粘贴图像。复制/粘贴或拖放图像到提示中。
  • 提供 URL 用于文档和 API 参考。使用 /permissions 来允许列表经常使用的域。
  • 管道数据 通过运行 cat error.log | claude 直接发送文件内容。
  • 让 Claude 获取它需要的东西。告诉 Claude 使用 Bash 命令、MCP 工具或通过读取文件来自己拉取上下文。

给 Claude 采访你的机会

对于更大的功能，让 Claude 先采访你。从最小的提示开始，要求 Claude 使用 AskUserQuestion 工具采访你。

对于更大的功能，让 Claude 先采访你。从最小的提示开始，采访完成，启动新会话来执行它。新会话有干净的上下文，完全专注于实现。

先探索，再规划，最后编码

将研究和规划与实现分开，以避免解决错误的问题。

让 Claude 直接编码可能会产生错误的代码。使用 [Plan Mode] 将探索与执行分开。
推荐的工作流有四个阶段：

  1. 探索：进入 Plan Mode。Claude 读取文件并回答问题，不进行任何更改
  2. 规划：要求 Claude 创建详细的实现计划
  3. 实现：切换出 Plan Mode 并让 Claude 编码，根据其计划进行验证
  4. 提交：要求 Claude 使用描述性消息进行提交并创建 PR

Plan Mode 很有用，但也增加了开销。

  • 对于范围明确且修复很小的任务（如修复拼写错误、添加日志行或重命名变量），要求 Claude 直接执行。
  • 当你对方法不确定、更改修改多个文件或你不熟悉被修改的代码时，规划最有用。

搞好环境

编写有效的 CLAUDE.md

运行 /init 根据你的当前项目结构生成启动 CLAUDE.md 文件，然后随时间精化。

  • CLAUDE.md 是一个特殊文件，Claude 在每次对话开始时读取。包括 Bash 命令、代码风格和工作流规则。这给 Claude 提供了它无法从代码中推断的永久上下文。
  • CLAUDE.md 文件没有必需的格式，但保持简短和易读。
  • CLAUDE.md 在每个会话中加载，所以只包括广泛适用的东西。对于仅有时相关的域知识或工作流，改用 skills。Claude 按需加载skill，不会使每次对话都加载。
  • 保持简洁。对于每一行，问自己：“删除这个会导致 Claude 犯错吗？” 如果不会，删除它。
  • 膨胀的 CLAUDE.md 文件会导致 Claude 忽略你的实际指令！
✅ 包括 ❌ 排除
Claude 无法猜测的 Bash 命令 Claude 可以通过读取代码弄清楚的任何东西
与默认值不同的代码风格规则 Claude 已经知道的标准语言约定
测试指令和首选测试运行器 详细的 API 文档（改为链接到文档）
存储库礼仪（分支命名、PR 约定） 经常变化的信息
特定于你的项目的架构决策 长解释或教程
开发者环境怪癖（必需的环境变量） 自明的实践，如"编写干净的代码"
常见陷阱或非显而易见的行为 文件逐个描述代码库

配置权限

使用 auto mode 让分类器处理批准，使用 /permissions 来允许列表特定命令，或使用 /sandbox 进行操作系统级隔离。每种方式都减少中断，同时让你保持控制。

默认情况下，Claude Code 会经常停下来，寻求你的权限：文件写入、Bash 命令、MCP 工具等。安全但繁琐。有三种方式来减少这些中断：

  1. Auto mode：一个单独的分类器模型审查命令并仅阻止看起来有风险的东西：范围升级、未知基础设施或由敌对内容驱动的操作。最适合当你信任任务的总体方向但不想点击通过每一步时
  2. 权限允许列表：允许你知道是安全的特定工具，如 npm run lint 或 git commit
  3. 沙箱：启用操作系统级隔离，限制文件系统和网络访问，允许 Claude 在定义的边界内更自由地工作

阅读更多关于 权限模式、权限规则 和 沙箱。

使用 CLI 工具

告诉 Claude Code 在与外部服务交互时使用 CLI 工具，如 gh、aws、gcloud 和 sentry-cli。

告诉 Claude Code 在与外部服务交互时使用 CLI 工具，如 gh、aws、gcloud 和 sentry-cli。

连接 MCP 服务器

运行 claude mcp add 来连接外部工具，如 Notion、Figma 或你的数据库。

使用 MCP你可以要求 Claude 从问题跟踪器实现功能、查询数据库、分析监控数据、集成来自 Figma 的设计并自动化工作流。

设置 hooks

使用 hooks 来处理必须每次发生且没有例外的操作。

使用 hooks 来处理必须每次发生且没有例外的操作。在 Claude 工作流中的特定点自动运行脚本。与 CLAUDE.md 指令不同，hooks 是确定性的，保证操作发生。

创建 skills

在 .claude/skills/ 中创建 SKILL.md 文件，为 Claude 提供域知识和可重用工作流。

使用特定于你的项目、团队或域的信息扩展 Claude 的skill。Claude 在相关时自动应用它们，或者你可以使用 /skill-name 直接调用它们。

创建自定义 subagents

在 .claude/agents/ 中定义专门的助手，Claude 可以委托给它们来处理隔离的任务。

Subagents在自己的 context 中运行，拥有自己的一组允许的工具。它们对于读取许多文件或需要专门关注而不会使你的主对话混乱的任务很有用。

安装 plugins

运行 /plugin 来浏览市场。Plugins 添加 skills、工具和集成，无需配置。

Plugins将 skills、hooks、subagents 和 MCP 服务器捆绑成一个可安装单元中。
有关在 skills、subagents、hooks 和 MCP 之间选择怎么选，请参阅 扩展 Claude Code。

管理会话

尽早且经常改正方向

一旦你注意到 Claude 偏离轨道，立即改正它。

如果你在一个会话中对同一问题改正了 Claude 两次以上，上下文就充满了失败的方法。运行 /clear 并使用更具体的提示重新开始。干净的会话与更好的提示几乎总是优于长会话与累积的改正。

积极管理上下文

在不相关的任务之间频繁运行 /clear 来重置 context。

在长会话中，Claude 的上下文可能会充满无关的对话、文件内容和命令。这会分散 Claude 的注意力。且Claude Code 在接近最大上下文限制时自动压缩对话历史，释放上下文空间，这保留了重要的代码和决策，但会丧失细节。

  • 在不同的任务之间频繁使用 /clear 来完全重置上下文
  • 仅压缩对话的一部分时，请使用 Esc + Esc 或 /rewind，选择消息检查点，并选择 从这里总结 或 总结到这里。请参阅 恢复与总结。
  • 在 CLAUDE.md 中使用像 "When compacting, always preserve the full list of modified files and any test commands" 这样的指令来自定义压缩行为，以确保关键上下文在压缩后仍能保留。
  • 对于不需要留在上下文中的快速问题，使用 /btw。答案就永远不会进入对话历史也不增加上下文。

使用 subagents 进行调查

使用 “use subagents to investigate X” 委托研究。它们在单独的 context 中探索，为实现保持你的主对话干净。

由于存在上下文约束，所以 subagents 是可用的最强大的工具之一。当 Claude 研究代码库时，它读取许多文件，所有这些都消耗你的主对话的上下文。Subagents 在单独的上下文中运行并报告摘要：

subagent 探索代码库、读取相关文件并报告发现，所有这些都不会使你的主对话混乱。

使用 rewind 进行回退

Claude 进行的每个提示都会创建一个检查点。你可以将对话、代码或两者恢复到任何之前的检查点。

Claude 在每次更改前自动对文件进行快照，以便检查点可以恢复它们。双击 Escape 或运行 /rewind 来打开 rewind 菜单。你可以：

  • 仅恢复对话
  • 仅恢复代码
  • 恢复两者
  • 从选定的消息进行总结

有关详细信息，请参阅 检查点。
检查点在会话中永远可用，所以你可以恢复之前的会话并稍后 rewind。

注意：检查点仅跟踪 Claude 进行的更改，不跟踪外部进程。这不是 git 的替代品。

恢复对话

使用 /rename 给会话命名，并像对待分支一样对待它们：每个工作流都有自己的持久 context。

Claude Code 在本地保存对话，运行 claude --continue 来继续最近的会话，运行 claude --resume 来从列表中选择。有关完整的恢复、分支和命名控制集，请参阅 管理会话。

常见的错误用法

这些是常见的错误。尽早识别它们可以节省时间：

  • 充满无效信息的会话。 你从一个任务开始，然后问 Claude 一些不相关的东西，然后回到第一个任务。Context 充满了无关的信息。

修复：在不相关的任务之间 /clear。

  • 一次又一次地改正。 Claude 做错了什么，你改正它，它仍然是错的，你再改正。Context 被失败的方法污染。

修复：在两次失败的改正后，/clear 并编写一个更好的初始提示，包含你学到的东西。

  • 过度指定的 CLAUDE.md。 如果你的 CLAUDE.md 太长，Claude 会忽略一半，因为重要的规则在噪音中丢失。

修复：无情地修剪。如果 Claude 已经在没有指令的情况下正确地做某事，删除它或将其转换为 hook。

  • 没有验收标准的工作。 Claude 产生一个看起来合理的实现，但不处理边界情况。

修复：始终提供验证（测试、脚本、屏幕截图）。如果你不能验证它，不要发布它。

  • 不限定边界的探索。 你要求 Claude "调查"某些东西而不限定范围。Claude 读取数百个文件，填充 context。

修复：狭隘地限定调查或使用 subagents，以便探索不会消耗你的主 context。
