C++基础：异常处理

异常处理概述

什么是异常

  • 异常的定义：C++ 中的异常是程序运行时发生的非预期错误（如除 0 运算、空指针访问、文件打开失败、数组越界等），区别于编译错误（语法问题）和逻辑错误（代码逻辑不符合预期但语法合法）。

异常处理的目标

  • 错误隔离：正常代码和错误处理代码分开，提升代码可读性和维护性；
  • 程序健壮性：出错后不直接崩溃，可捕获异常并执行兜底逻辑（如释放资源、记录日志、提示用户）；
  • 错误信息可追溯：通过异常对象携带详细错误信息，便于定位问题。

C++ 异常处理是一把 “双刃剑”—— 它提供了优雅的错误处理能力，但也存在显著的门槛和局限性，以下是优缺点的全面分析：

为什么需要异常

  • 错误逻辑与业务逻辑分离：异常将 “出错处理” 从 “正常执行” 的代码中抽离，避免了层层嵌套的返回值检查，代码可读性和维护性大幅提升。
  • 错误可跨多层函数传递：异常无需在每一层函数中显式传递错误，可直接从深层函数抛到顶层处理，解决了 “错误传递链断裂” 的问题。
  • 错误信息更丰富：异常对象（如自定义异常类）可携带错误码、文件名、行号、详细描述等信息，远优于单一的返回值（如-1）。
  • 适配 RAII 实现异常安全：结合 RAII（如std::unique_ptr、std::lock_guard），栈展开时会自动销毁局部对象，避免资源泄漏（返回值方式易遗漏资源释放）。

为什么异常被 “诟病”

  • 性能开销（仅在抛出时）：异常本身是 “不常用路径” 设计，不抛异常时几乎无开销，但抛出异常时的栈展开（销毁局部对象、回溯调用栈）有明显开销（比返回值高 1-2 个数量级），不适合高频错误场景。
  • 执行流非线性，调试成本高：异常会跳转到任意上层的 catch 块，执行流不再是 “从上到下” 的线性逻辑，新手难以追踪错误传播路径，调试时需额外关注异常抛出点。
  • 异常安全门槛高：编写 “异常安全” 的代码（如保证资源不泄漏、对象状态一致）需要深入理解栈展开、析构函数规则，新手易写出有隐患的代码（如构造函数抛异常导致对象未完全初始化）。
  • 兼容性与环境限制：
    • 与 C 语言不兼容：C 语言无异常机制，C++ 代码抛异常到 C 代码中会直接崩溃；
    • 部分环境禁用：嵌入式系统、内核开发、高性能游戏引擎（如 Unreal Engine 早期）等场景，编译器 / 运行时可能禁用异常（减少二进制体积、提升稳定性）。
  • 滥用风险高：新手易滥用catch(...)捕获所有异常却不处理 / 不重新抛出，导致错误被隐藏，排查难度大幅增加。

为什么 return 更常用

实际开发中返回值（错误码、std::optional、std::pair等）仍是主流，原因如下：

  1. 场景适配性：简单错误用返回值更轻量

大部分开发中的错误是 “简单且高频” 的（如参数校验、边界检查、状态判断），这类错误用返回值处理更直接，无需引入异常的复杂度：

  1. 团队 / 项目规范：很多场景禁用异常
  • 老项目兼容：大量 C++ 老项目（如 Linux 内核周边工具、早期开源库）为了兼容 C 代码或简化逻辑，全程禁用异常；
  • 大厂规范：Google、Facebook 等公司的 C++ 风格指南，对 “非关键错误” 推荐返回值，仅允许在 “不可恢复错误”（如内存分配失败）时用异常；
  • 统一风格：团队协作时，返回值的错误处理方式更易统一（异常的使用规则难标准化）。
  1. 性能敏感场景：避免异常的栈展开开销

在高频调用的函数中（如游戏帧更新、高频交易系统、数据结构底层操作），即使异常抛出概率低，也会被禁用 —— 因为单次异常的开销可能导致帧掉卡、交易延迟超标。

  1. 调试和维护成本更低

返回值的错误处理是 “显式且线性” 的：调用函数时必须显式检查返回值，IDE 调试时可直接看到错误码，新人接手项目时无需理解复杂的异常传播路径。

  1. 现代替代方案的出现

C++23 引入的std::expected（类似 Boost.Outcome），结合了返回值和异常的优点：既保留了返回值的 “线性执行流”，又能携带丰富的错误信息，成为越来越多项目的首选，进一步降低了异常的使用场景。

三、异常与返回值的合理选择原则

实践中并非 “非此即彼”，而是按场景选择：

场景类型 推荐方式 示例
简单高频错误（参数校验） 返回值 /std::optional 字符串判空、数组越界检查
不可恢复错误（资源失败） 异常 内存分配失败、文件打开失败
跨多层函数的错误 异常 网络请求深层错误
性能敏感 / 高频调用函数 返回值 /std::expected 游戏帧更新、数据结构操作
C/C++ 混编场景 返回值 调用 C 语言函数的封装层

异常处理核心语法

try-catch

  • 基本结构：
  • 执行流程：
    1. 执行try块内的代码；
    2. 如果try块内没有抛出异常，跳过所有catch块，执行后续代码；
    3. 如果try块内抛出异常，立即终止try块执行，从上到下匹配catch块的异常类型：
      • 找到匹配的catch：执行该catch的处理逻辑，然后执行catch后的代码；
      • 未找到匹配的catch：程序调用std::terminate()终止运行。

throw 关键字

  • 基本语法：throw 异常对象/值;（可抛出任意类型：内置类型、字符串、自定义类对象等，推荐抛异常类对象）。
  • 注意事项：
    1. throw可以在函数内任意位置使用，抛出后当前函数立即终止执行；
    2. 空throw（throw;）只能在catch块内使用，作用是重新抛出当前捕获的异常（用于多层处理，如先记录日志，再向上传递）；
    3. 避免抛基础类型（如int、string），难以统一管理和扩展，推荐使用标准库 / 自定义异常类。

  • 核心方法：virtual const char* what() const noexcept;（返回异常的描述信息，子类需重写）；

用户定义的异常

设计原则

为了和标准异常体系兼容，自定义异常需遵循以下原则：

  1. 继承std::exception或其派生类（如std::runtime_error）；
  2. 重写what()方法（返回 const char * 类型的错误信息，且声明为noexcept）；
  3. 按需添加自定义成员（如错误码、文件名、行号等），提升错误信息的可读性。

noexcept 关键字

核心作用

noexcept是 C++11 引入的关键字，用于声明函数不会抛出异常（或在特定条件下不抛出）：

  • 语法 1：void func() noexcept; → 函数绝对不会抛出异常；
  • 语法 2：void func() noexcept(表达式); → 表达式为 true 时，函数不抛异常（如noexcept(is_trivially_copyable<T>::value)）。

使用场景

  1. 析构函数：C++11 后析构函数默认noexcept(true)，如果手动声明为noexcept(false)且抛异常，会导致程序终止；
  2. 移动构造 / 移动赋值函数：声明noexcept可让容器（如vector）优先使用移动语义（而非拷贝），提升性能；
  3. 性能敏感的函数：编译器对noexcept函数有优化空间（无需生成异常处理的额外代码）；
  4. 明确异常行为的函数：向调用者清晰声明 “该函数不会抛异常”，减少调用者的异常处理成本。

与异常的关系

  • 如果声明noexcept的函数实际抛出了异常，程序会直接调用std::terminate()终止，不会展开调用栈；
  • noexcept是编译期声明，不是运行时检查（编译器不会阻止函数抛异常，只是抛异常后程序直接终止）；
  • noexcept不能替代异常处理，仅用于明确函数的异常行为。

异常与调用栈

异常抛出时的调用栈展开（stack unwinding）

  • 定义：当异常抛出且当前函数没有匹配的catch块时，编译器会沿着函数调用栈向上回溯（从抛异常的函数→调用该函数的上层函数→…→main 函数），这个过程称为 “栈展开”。
  • 栈展开的核心行为：回溯过程中，会依次销毁栈上的局部对象（调用其析构函数），直到找到匹配的catch块；如果到main函数仍未找到，调用std::terminate()。

调用栈展开的规则与注意事项

  1. 析构函数必须安全：栈展开时会调用局部对象的析构函数，如果析构函数抛出异常，会导致std::terminate()（因为栈展开过程中无法处理新的异常）；
  2. 动态分配的资源需手动释放：栈展开只会销毁栈上的局部对象，堆上的动态资源（如new分配的内存）需手动释放（或用 RAII，如std::unique_ptr）；
  3. catch 块的匹配顺序：栈展开找到第一个匹配的catch后就会停止，因此多个 catch 需按 “子类→基类” 顺序排列；
  4. noexcept 函数中断栈展开：如果栈展开过程中进入声明noexcept的函数且该函数抛异常，栈展开会立即停止，程序终止。

异常处理实践

异常捕获的原则

  1. 按 “子类→基类” 顺序捕获：先捕获具体的派生异常（如FileOpenError），再捕获基类异常（如exception），否则基类 catch 会覆盖子类 catch；cpp运行
  1. 避免滥用 catch (…)：catch(...)会捕获所有异常，但无法获取异常信息，仅建议作为 “兜底” 且必须重新抛出（否则隐藏错误）；
  2. 只捕获能处理的异常：如果当前函数无法处理某个异常，不要捕获，让上层函数处理（避免 “捕获后不处理也不重新抛出”）。

异常安全（资源管理、noexcept 的合理使用）

  1. RAII 管理资源：使用 “资源获取即初始化”（如std::unique_ptr、std::lock_guard）管理动态内存、锁、文件句柄等，栈展开时会自动调用析构函数释放资源，避免泄漏；cpp运行
  1. 析构函数必须 noexcept：默认情况下析构函数是noexcept(true)，不要手动改为noexcept(false)；
  2. 移动语义优先声明 noexcept：容器（如vector）在扩容时，若元素的移动构造是noexcept，会用移动而非拷贝，提升性能且更安全。

常见坑点

  1. 析构函数抛异常：栈展开时析构函数抛异常会导致程序终止，绝对禁止；
  2. 空 throw 在非 catch 中使用：throw;只能在 catch 块内使用，非 catch 中使用会导致未定义行为；
  3. 抛基础类型（int/string）：难以统一管理，无法扩展错误信息，推荐抛异常类；
  4. 捕获异常后忽略：catch(...) { }会隐藏所有错误，导致程序 “看起来正常” 但实际出错，必须避免；
  5. 在构造函数中抛异常：构造函数抛异常时，对象未完全构造，析构函数不会被调用，需确保构造过程中分配的资源已释放。

GDB调试

简介

GNU项目调试器（GNU project DeBugger，GDB）是一个强大的调试器，详见https://www.gnu.org/software/gdb/或https://sourceware.org/gdb/。我们可以使用命令行与GDB交互。

Clang也有一个很好的调试器，叫作LLDB（Low Level DeBugger），详见https://lldb.llvm.org/。它与GDB命令兼容。
