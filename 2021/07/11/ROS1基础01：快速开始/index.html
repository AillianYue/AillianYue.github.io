ROS1基础01：开始

本篇定位：如果你已有一定的 Linux / C++ 基础，想快速上手 ROS1 并用起来，而不是从零系统学完官方教程，那这篇就是为你准备的。

学前须知

请不要系统学习 ROS

ROS 本质上是一个工具集：它帮你解决进程间通信、数据记录、可视化等工程问题。就像你不会先通读《电动螺丝刀从入门到精通》再拧螺丝一样，学习 ROS 最有效的方式也是带着问题去实践：

  • 我想让两个程序互相发数据？→ 学 Topic + Publisher / Subscriber
  • 我想看看谁在发什么数据？→ 学 rostopic list / echo
  • 我想把传感器数据存下来回放？→ 学 rosbag

如果反过来，先从头到尾啃完官方 Wiki 再动手，反而容易迷失在庞杂的概念里，学完就忘。

所以本节的策略是：先花最少的时间，掌握 ROS 中最基础、最常用的一小部分知识（节点、话题、消息、常用命令行），然后就直接去项目里"用起来"。遇到不会的，查文档、查这篇、搜一下，都比一次学完所有内容更高效。

当然，如果你已经在实践中遇到瓶颈，回头系统学习也是必要的，但那是后话，不是第一步。

ROS 架构简介

ROS1（Robot Operating System 1）采用一种去中心化的点对点通信架构，核心组件包括：

组件 作用
roscore ROS 的"管家"：启动 Master（名称服务）、Parameter Server、日志节点。任何 ROS 程序运行前必须先启动 roscore
Master 名称服务器：让节点之间互相发现。节点启动时向 Master 注册自己的名字和话题信息
Node（节点） 一个可执行程序，是 ROS 中的最小计算单元。一个机器人系统通常由几十上百个节点组成
Topic（话题） 节点间通信的异步总线。发布者往话题上发消息，订阅者从话题上收消息，双方互不了解对方的存在
Message（消息） 通信的数据结构。可以是 int、float、string 等基本类型，也可以是嵌套的自定义类型
Service（服务） 同步的请求-响应通信（本篇暂不展开，后续进阶）

为什么要用 ROS

这里只列三个最直接的理由：

  1. 模块化：你不必把感知、规划、控制写在一个巨型 main() 里。拆成多个节点，各管各的，出问题也好定位。
  2. 复用生态：ROS 社区有大量现成的包，驱动（摄像头、激光雷达）、导航栈（move_base）、SLAM（gmapping、cartographer）、规划（OMPL）。站在巨人肩膀上。
  3. 工具链：rosbag 录数据 → rviz 可视化 → rqt 调试 → rosnode/rostopic 命令行诊断。一套工具覆盖从开发到 Debug 的完整链路。

核心概念

节点 (Node)

节点是 ROS 中的可执行进程。每个节点有一个唯一的名称（如 /talker、/listener），通过名称在 Master 上注册。

关键特征：

  • 一个节点就是一个进程（可执行文件），通常对应一个类或一组功能
  • 节点之间彼此独立：一个节点崩溃不会影响其他节点
  • 节点通过 Topic、Service、Action 等方式通信

话题 (Topic)

话题是 ROS 中最重要的通信机制：异步、多对多的发布-订阅模型。

  • 发布者把消息"扔"到话题上
  • 订阅者从话题上"收"消息
  • 双方解耦：不知道对方的存在
  • 话题名称是一个全局字符串，如 /odom、/camera/image_raw

命名规则：

  • 以 / 开头表示全局名称，如 /camera/rgb/image_raw
  • 不以 / 开头表示相对名称，相对于节点的命名空间
  • 私有名称以 ~ 开头，如 ~param_name

一个话题可以有多个发布者和多个订阅者。

消息 (Message)

消息是 ROS 节点间通信的数据结构。类似 C 语言中的 struct，由一组类型化的字段组成。

标准消息类型（在 std_msgs 包中）：

类型 C++ 对应
std_msgs::Int32 int32_t
std_msgs::Float64 double
std_msgs::String std::string
std_msgs::Header 包含 timestamp 和 frame_id 的公共头部

常用复合消息：

包名 消息类型 用途
geometry_msgs Twist 速度指令（线速度 + 角速度）
geometry_msgs Pose / PoseStamped 位置 + 朝向
sensor_msgs Image 图像数据
sensor_msgs LaserScan 激光雷达扫描数据
nav_msgs Odometry 里程计数据
tf2_msgs TFMessage 坐标变换

发布者 (Publisher)

发布者节点负责向话题发送消息。创建流程：

  1. 通过 NodeHandle::advertise<T>() 告诉 Master"我要发某种消息了"
  2. 用 publish(msg) 发送消息
  3. 通常配合定时器或传感器回调循环发送

订阅者 (Subscriber)

订阅者节点接收话题上的消息。核心机制是回调函数，消息到达时，ROS 自动调用你注册的回调。

创建流程：

  1. 通过 NodeHandle::subscribe<T>() 告诉 Master"我要订阅某种消息"
  2. 注册一个回调函数，每次消息到达时自动调用
  3. 调用 ros::spin() 或 ros::spinOnce() 让回调被触发

重要区分：

  • ros::spin() — 阻塞，直到节点关闭。适合纯订阅者。
  • ros::spinOnce() — 非阻塞，处理一次待处理回调后立即返回。适合在 while 循环中既要发布又要接收的场景（见上面 talker 的写法）。
  • 如果一个节点既发布又订阅，通常在 while 循环中调 spinOnce()。

回调函数机制

订阅者通常基于回调函数，这是一种异步模式：你注册好回调，ROS 内部线程在消息到达时自动调用它，主线程继续做其他事情或者 spin() 等待。

🔑 回调陷阱提醒：

  • 回调函数中不要做阻塞操作（如 sleep、长耗时计算），它会阻塞整个节点处理其他回调。
  • 如果需要在回调里做耗时操作，把数据拷贝出来放到一个队列/缓冲区，在主循环中处理。
  • 消息指针是 ConstPtr，你不能修改它。要改就拷贝再改。

Rosbag 录制与回放

rosbag 是 ROS 中最强大的工具之一，它可以把当前运行的话题数据录制到磁盘，然后离线回放。这在算法调试、复现 bug、数据集采集中极为常用。

Rqt 可视化工具

rqt 是 ROS 的 Qt 图形化工具套件。它以一个插件式界面替代了早期零散的单用途 GUI 工具。

启动

启动后，通过顶部菜单栏 Plugins → 展开对应分类 → 点击需要的插件，即可在界面中手动添加各种面板。所有插件都在一个窗口里，可以自由拖拽排列。

当然，如果你已经知道要开哪个插件，也可以直接用命令行一步到位：rqt_publisher（手动发消息）、rqt_topic（话题监控）、rqt_graph（节点关系图）、rqt_plot（实时曲线）、rqt_console（日志）、rqt_image_view（图像）、rqt_reconfigure（动态参数）等。

最常用场景

场景 操作方式
我想手动往话题上发一条消息 Plugins → Topics → Message Publisher → 选话题 → 填字段值 → Publish
我想查看话题的消息内容和频率 Plugins → Topics → Topic Monitor → 选话题即可实时查看
我想看节点谁连接谁 Plugins → Introspection → Node Graph
我想看某个值的实时变化趋势 Plugins → Visualization → Plot → 选择话题字段
我想看摄像头画面 Plugins → Visualization → Image View → 选 /camera/image_raw
ROS_INFO 太多刷屏了 Plugins → Logging → Console → 按级别过滤
我想在运行时调参数 Plugins → Configuration → Dynamic Reconfigure

总结

在 ROS1 中最核心、最高频的知识点：

概念 一句话
roscore ROS 的管家，跑任何东西前必须先启动它
ros::init 初始化一个节点
NodeHandle 节点和 ROS 的"门把手"，用它创建发布者/订阅者
Publisher 往话题上发消息
Subscriber 从话题上收消息（通过回调函数）
ros::spin() 阻塞等待回调
ros::spinOnce() 非阻塞，处理一次回调
rostopic echo 查看话题数据（最常用的调试命令）
rosbag 录数据 & 回放数据

学习建议：

  1. 搭建一个 ROS1 工作环境（Ubuntu 20.04 + ROS Noetic，或 Docker）
  2. 照着本文的代码段写一个 talker + listener，跑通
  3. 在实际项目中遇到需求时，带着问题去：需要坐标变换学 tf2，需要动作控制学 Action，需要参数服务器学 Parameter Server
  4. 从 ROS1 过渡到 ROS2 时，概念相通，API 类似，迁移成本不高

ROS 只是一个工具，能帮你解决问题的工具才是好工具。不用成为 ROS 专家，够用就行。
