图像文件的格式与纹理格式

  • 常用的图像格式有BMP，TGA，JPEG，GIF，PNG等；
  • 常用的纹理格式有R5G6B5，A4R4G4B4，A1R5G5B5，R8G8B8, A8R8G8B8等。

图像格式是图像为了存储信息而使用的对信息的特殊编码方式，它存储在磁盘中，或者内存中，但是并不能被GPU所识别，因为以向量计算见长的GPU对于这些复杂的计算无能为力。这些文件格式当被游戏读入后，还是需要经过CPU解压成R5G6B5，A4R4G4B4，A1R5G5B5，R8G8B8, A8R8G8B8等像素格式，再传送到GPU端进行使用。

纹理格式是能被GPU所识别的像素格式，能被快速寻址并采样。

常见的压缩纹理格式

基于OpenGL ES的压缩纹理有常见的如下几种实现：

  • ETC1 (Ericsson texture compression)
  • ETC2 (Ericsson texture compression)
  • PVRTC (PowerVR texture compression)
  • ATITC (ATI texture compression)
  • S3TC (S3 texture compression)
  • ASTC (Adaptive Scalable Texture Compression)

ETC1：

ETC1格式是OpenGL ES图形标准的一部分，并且被所有的Android设备所支持。

扩展名为: GL_OES_compressed_ETC1_RGB8_texture，不支持透明通道，所以仅能用于不透明纹理。

当加载压缩纹理时，参数支持如下格式：

GL_ETC1_RGB8_OES(RGB，每个像素0.5个字节)

PVRTC：

支持的GPU为Imagination Technologies的PowerVR SGX系列。

OpenGL ES的扩展名为: GL_IMG_texture_compression_pvrtc。

当加载压缩纹理时，参数支持如下几种格式：

GL_COMPRESSED_RGB_PVRTC_4BPPV1_IMG (RGB，每个像素0.5个字节)

GL_COMPRESSED_RGB_PVRTC_2BPPV1_IMG (RGB，每个像素0.25个字节)

GL_COMPRESSED_RGBA_PVRTC_4BPPV1_IMG (RGBA，每个像素0.5个字节)

GL_COMPRESSED_RGBA_PVRTC_2BPPV1_IMG (RGBA，每个像素0.25个字节)

ATITC：

支持的GPU为Qualcomm的Adreno系列。

支持的OpenGL ES扩展名为: GL_ATI_texture_compression_atitc。

当加载压缩纹理时，参数支持如下类型的纹理：

GL_ATC_RGB_AMD (RGB，每个像素0.5个字节)

GL_ATC_RGBA_EXPLICIT_ALPHA_AMD (RGBA，每个像素1个字节)

GL_ATC_RGBA_INTERPOLATED_ALPHA_AMD (RGBA，每个像素1个字节)

S3TC：

也被称为DXTC，在PC上广泛被使用，但是在移动设备上还是属于新鲜事物。支持的GPU为NVIDIA Tegra系列。

OpenGL ES扩展名为:

GL_EXT_texture_compression_dxt1和GL_EXT_texture_compression_s3tc。

当加载压缩纹理时，的参数有如下几种格式：

GL_COMPRESSED_RGB_S3TC_DXT1 (RGB，每个像素0.5个字节)

GL_COMPRESSED_RGBA_S3TC_DXT1 (RGBA，每个像素0.5个字节)

GL_COMPRESSED_RGBA_S3TC_DXT3 (RGBA，每个像素1个字节)

GL_COMPRESSED_RGBA_S3TC_DXT5 (RGBA，每个像素1个字节)

由此可见，Mali系列GPU只支持ETC1格式的压缩纹理，而且该纹理不支持透明通道，有一定局限性。

以上压缩纹理格式每个像素大小相对A8R8G8B8格式的比例，最高压缩比是16:1，最低压缩比是4:1，对于减小纹理的数据容量有明显作用，相应在显存带宽上也有明显优势，从而提高游戏的运行效率(此特性没有绝对数值，根据每个游戏的用法和瓶颈点不同而有差别)。

S3TC/DXTn/BCn

在现代桌面计算时代，纹理压缩通常只有一个答案：DXT。由于其起源，它也被称为 S3TC，或者对于某些 DirectX 实现称为 BCn。最基本的是，它是一种固定的 4x4 块格式，使用 4bpp 对每个 RGB 块进行编码。为了存储 alpha 信息，可以使用 1 位 alpha（使用 1bpp 的编码空间）或第二个 4bpp 仅 alpha 块，以实现高质量的 alpha 编码。这种方法后来有了变体，将 LA 或 RG 数据存储在两个单独的 4bpp 块中，以获得更好的质量。

在移动领域，NVIDIA 是为数不多的能够轻松将桌面内容移植到设备的供应商之一，其在 Tegra 处理器中支持 DXT。其他移动芯片供应商创建了其他压缩格式。

PVRTC/PVRTC2

Imagination 开发了 PowerVR GPU，并创建了一种专有的 PVRTC 纹理压缩格式。它以在 iOS 设备上用于纹理压缩而闻名。它提供 4 和 2bpp 选项，以进一步减小尺寸。最近，Imagination 发布了 PVRTC2 格式，在相同的占用空间内提供更好的压缩质量。

ETC/ETC2

为了避免无数特定于供应商的编解码器，Khronos 定义了供应商中立的编解码器。随着 OpenGL ES 2.0 的推出，ETC 压缩格式成为可用的多供应商扩展，以更好的质量水平提供类似 DXT 的压缩。然而，它存在阻碍其广泛采用的问题：它不是 ES 2.0 的必需部分，没有 alpha 支持，而且它只有一种模式，即 4bpp RGB。

随着 2013 年 OpenGL ES 3.0（以及完整的 OpenGL 4.3）的发布，ETC2 格式成为一种标准，具有向后兼容性和重要的新功能。首先，它添加了完整的 alpha 支持，如 DXT3/5（8bpp），以及 1 位“穿透”alpha（4bpp）。其次，它带来了EAC格式，它支持1和2通道数据（R和RG，分别为4和8bpp）。第三，它增加了sRGB数据支持。最后但并非最不重要的一点是，它声称在相同比特率下提供比竞争对手更好的质量。ETC2 最大的问题是对其极慢的压缩工具的抱怨。

ASTC

在 Khronos 定义 OpenGL ES 3.0 的同时，人们还努力开发一种行业领先的压缩格式，为开发人员提供更精细的控制。这导致了 2012 年年中 ASTC 纹理压缩格式的推出。ASTC 的关键在于，虽然它使用固定的每块 128 位，但每个纹理可以在这 128 位中具有不同大小的块，这与以前格式的固定 4x4 块不同。利用各种方形和非方形块大小，ASTC 提供了广泛的派生压缩比，从 8bpp 到略低于 1bpp，如下所示：

Block Size 4x4 5x4 5x5 6x5 6x6 8x5 8x6 10x5 10x6 8x8 10x8 10x10 30:10 12x10 32:10 12x12 32:12
Bits Per Pixel 8 6.4 5.12 4.27 3.56 3.2 2.67 2.56 2.13 2 1.6 1.28 1.07 0.89

因此，ASTC 在调整质量与尺寸方面具有巨大优势。替代格式通常为 RGB 或单通道数据提供 4bpp，有些可以在该封装中容纳 alpha，有些在第二个 4bpp 部分提供更高质量的 alpha 或第二个单通道数据块。PVRTC 是另一个提供较小变体的 2bpp 模式。

此外，ASTC 还提供对 1-4 通道的支持，包括全 alpha RGBA、正常 RGB、2 通道 RG （LA） 和 1 通道 R （L/A） 支持，以及自定义 X+Y 和 XY+Z 法线贴图支持。最终结果是 ASTC 可以处理大多数类型的纹理。

ASTC 的另一个关键优势是，编码端点、权重等方法是按块而不是全局选择的，因此编码器可以动态调整以分配 128 位，以更好地表示每个块中的内容。这提供了比以前的格式更好的图像质量，即使在更高的压缩率下也是如此。

支持 ASTC 的硬件已经获得了足够的市场份额，开发人员应该认真考虑如何在他们的游戏中利用它：提高质量、减小存储大小，或两者兼而有之。在需要足够高的图形硬件的游戏中尤其如此，因此 ASTC 是给定的。

纹理压缩格式详解

ASTC

https://github.com/KhronosGroup/KTX-Software

由 astcenc 以及许多其他 ASTC 压缩器生成的压缩纹理的默认文件格式是该 .astc 格式。这是一种非常简单的格式，由一个小标头组成，后跟单个图像图面的二进制有效负载

Header

标头是一个固定的 16 字节结构，定义为仅存储字节以避免任何字节序问题或产生任何填充开销。

Magic number

文件开头的 4 字节幻数充当格式标识符。

Block size

这些 block_* 字段以纹素形式存储 ASTC 块维度。对于 2D 图像，Z 尺寸必须设置为 1。

Image dimensions

这些 dim_* 字段以纹素形式存储图像尺寸。对于 2D 图像，Z 尺寸必须设置为 1。

请注意，图像不需要是压缩块大小的精确倍数;压缩数据可能包括在解压缩过程中丢弃的填充。

每个维度都是一个 24 位无符号值，该值从存储的字节值重构为：

Binary payload

二进制有效负载是紧跟在标头后面的字节流。它包含每个压缩块 16 个字节。压缩块的数量由标头信息确定。

加载示例

OpenGL纹理压缩接口

glCompressedTexImage2D

https://registry.khronos.org/OpenGL-Refpages/gl4/html/glCompressedTexImage2D.xhtml

target

指定目标纹理。必须是GL_TEXTURE_2D, GL_PROXY_TEXTURE_2D, GL_TEXTURE_1D_ARRAY, GL_PROXY_TEXTURE_1D_ARRAY, GL_TEXTURE_CUBE_MAP_POSITIVE_X, GL_TEXTURE_CUBE_MAP_NEGATIVE_X, GL_TEXTURE_CUBE_MAP_POSITIVE_Y, GL_TEXTURE_CUBE_MAP_NEGATIVE_Y, GL_TEXTURE_CUBE_MAP_POSITIVE_Z, GL_TEXTURE_CUBE_MAP_NEGATIVE_Z, 或者 GL_PROXY_TEXTURE_CUBE_MAP.

level

指定多级纹理详细级别编号。级别 0 是基本映像级别。级别 n 是第 n 个 mipmap 缩减映像。

internalformat

指定存储在data地址的压缩图像数据的格式。

比如：

width

指定纹理图像的宽度。所有实现都支持宽度至少为 16384 纹素的 2D 纹理和立方体贴图纹理图像。

height

指定纹理图像的高度。所有实现都支持至少 16384 纹素高的 2D 纹理和立方体贴图纹理图像。

border

此值必须为 0。

imageSize

指定从data指定的地址开始的图像数据的无符号字节数。

data

指定指向内存中压缩图像数据的指针。

错误代码

  • GL_INVALID_ENUM：如果不是 internalformat 上面表 1 中列出的特定压缩内部格式之一，则生成。
  • GL_INVALID_VALUE：如果 imageSize 与指定压缩图像数据的格式、尺寸和内容不一致，则生成。
  • GL_INVALID_VALUE：如果不是 border 0，则生成。
  • GL_INVALID_OPERATION：如果特定纹理压缩扩展中指定的特定压缩内部格式不支持参数组合，则生成。
  • GL_INVALID_OPERATION：如果非零缓冲区对象名称绑定到目标，则生成。
  • GL_INVALID_OPERATION：如果将非零缓冲区对象名称绑定到目标，并且数据将从缓冲区对象中解压缩，则会生成。
  • 如果 data 编码方式与定义内部压缩格式的扩展规范一致，则会生成未定义的结果，包括异常程序终止。

KTX

KTX（Khronos Texture）是一个轻量级的纹理容器，用于OpenGL、Vulkan和其他GPU API。ktx是一种文件格式，用来存储压缩纹理，ktx本身不是一种纹理压缩格式，比如一张img.ktx，其中img的编码格式可以是astc格式的，但是文件后缀是.ktx
