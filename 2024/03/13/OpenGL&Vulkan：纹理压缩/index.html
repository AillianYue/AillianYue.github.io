纹理压缩概述

图像格式 vs 纹理格式

常见的图像格式如 BMP、JPEG、PNG 等是为磁盘存储而设计的编码方式。GPU 无法直接识别这些格式——需要 CPU 解码为 RGBA 等像素格式后才能上传到显存。

纹理格式则是 GPU 原生可识别的像素格式，能被采样器直接寻址。压缩纹理在 GPU 端保持压缩状态，无需 CPU 解压即可采样，大幅降低带宽压力和功耗。

为什么需要压缩纹理？

  • 降低显存占用（压缩比可达 4:1 ~ 12:1）
  • 减少带宽消耗，提高渲染性能
  • 降低发热和功耗（尤其移动设备）
  • 缩小文件体积，加快加载速度

常见压缩纹理格式

基于 OpenGL ES 生态的主流压缩纹理格式：

格式 开发者 特点
ETC1 Ericsson OpenGL ES 2.0 标准，全 Android 支持，不支持透明通道
ETC2 Khronos OpenGL ES 3.0 标准，向后兼容 ETC1，支持透明通道和 sRGB
PVRTC Imagination PowerVR GPU 原生，广泛用于 iOS，提供 2bpp/4bpp 选项
S3TC / DXTn / BCn S3 Graphics / Microsoft 桌面标准（DirectX），NVIDIA Tegra 支持
ATITC AMD / Qualcomm Adreno GPU 原生格式
ASTC Khronos 新一代标准，灵活可扩展的 block size（4x4 ~ 12x12），支持 1-4 通道，sRGB

ETC1：OpenGL ES 图形标准的一部分，被所有 Android 设备支持。扩展名 GL_OES_compressed_ETC1_RGB8_texture，仅支持 RGB，不支持透明通道。内部格式为 GL_ETC1_RGB8_OES（0.5 字节/像素）。

ETC2：随 OpenGL ES 3.0 发布的标准格式。在 ETC1 基础上增加了完整 alpha 支持（EAC，8bpp）和 1 位透穿 alpha（4bpp），同时支持 1/2 通道数据（R、RG）和 sRGB。向后兼容 ETC1。

PVRTC：Imagination PowerVR GPU 的专有格式，iOS 生态中的纹理压缩标准。提供 2bpp 和 4bpp 两种模式，支持 RGB 和 RGBA。

S3TC / DXTn / BCn：桌面端事实标准。基本格式为固定 4x4 块、4bpp 的 RGB 编码。Alpha 通过额外 4bpp 块存储（DXT3/5）。在 DirectX 生态中称为 BCn。

ASTC：Khronos 在 2012 年推出的行业领先格式。固定每块 128 位，但支持 14 种 block size（4x4 到 12x12），压缩比从 8bpp 到 ~0.89bpp 可调。支持 1-4 通道、sRGB、HDR。编码端点和权重按块动态选择，相同比特率下质量优于早期格式。

纹理格式选择指南

选择纹理格式需要平衡兼容性、性能和质量。以下按场景推荐：

场景 推荐格式 原因
现代 Android / 通用移动端 ASTC 4x4 ~ 8x8 兼容 ES 3.1+，质量/压缩比灵活可调，支持透明通道和 sRGB
兼容 OpenGL ES 2.0 设备 ETC1 Android 全覆盖，但不支持透明通道，需配合 Alpha 掩码
兼容 OpenGL ES 3.0+ 设备 ETC2 ES 3.0 强制要求，向后兼容 ETC1，支持 RGBA 和 sRGB
iOS 生态（旧设备） PVRTC 4bpp PowerVR GPU 原生，旧 iOS 设备首选
桌面端（DirectX） BC1~BC7 (BCn) DirectX 生态标准，驱动支持最好
HDR 内容 ASTC HDR / BC6H 移动端用 ASTC HDR，桌面端用 BC6H
UI 图标 / 开发调试 PNG（无压缩） 无损、通用，无需预处理管线

经验法则：能选 ASTC 就选 ASTC；需向后兼容退到 ETC2；桌面独占用 BCn；UI 小图留 PNG。

PVR 格式详解

PVR（PowerVR）是 Imagination Technologies 定义的纹理容器格式。

注意：PVR 是一个容器格式（类似 KTX），可以存储压缩纹理（PVRTC）也可以存储未压缩的 RGBA 数据。本文加载的是 PVR v3 容器中的未压缩 R8G8B8A8 数据，而非 PVRTC 压缩纹理。加载 PVRTC 压缩纹理需要使用 glCompressedTexImage2D 等专用 API。

PVR v3 头部结构（52 字节）：

偏移 大小 字段 说明
0 4 Version 版本标识，固定 0x03525650
8 8 Pixel Format 像素格式标识
16 4 Colour Space 颜色空间：0 = lRGB, 1 = sRGB
20 4 Channel Type 通道类型
24 4 Height 图像高度
28 4 Width 图像宽度
32 4 Depth 图像深度
36 4 Num Surfaces 表面数量
40 4 Num Faces 面数（立方体贴图）
44 4 MIP Map Count Mipmap 级别数
48 4 Meta Data Size 元数据大小

头部之后紧接元数据段，然后才是像素数据。非压缩格式在头部像素格式字段的前 4 字节会写入通道标识（如 rgba）和位深（如 8）。

PVR 格式加载

PVR 加载的核心是：验证魔数 → 解析头部字段 → 跳过元数据 → 提取像素数据。有了 ReadFileToBuffer，文件 I/O 的样板代码被剥离，函数只关心格式特有的解析逻辑。

ETC2 (KTX) 格式详解

KTX（Khronos Texture）是 Khronos 定义的轻量级纹理容器格式，用于 OpenGL / Vulkan 等 GPU API。KTX 本身不是一种压缩格式，而是容器——内部的纹理数据可以是 ETC2、ASTC 或其他格式。

KTX 头部结构（64 字节）：

偏移 大小 字段 说明
0 12 Identifier 文件标识 0xAB4B5458...
12 4 Endianness 字节序检测
16 4 glType 数据类型（压缩纹理为 0）
20 4 glTypeSize 数据类型大小
24 4 glFormat 像素格式（压缩纹理为 0）
28 4 glInternalFormat 内部格式（如 ETC2 的 0x9274 / 0x9279）
32 4 glBaseInternalFormat 基础内部格式
36 4 Pixel Width 图像宽度
40 4 Pixel Height 图像高度
44 4 Pixel Depth 图像深度
48 4 Number of Array Elements 数组元素数
52 4 Number of Faces 面数
56 4 Number of Mipmap Levels Mipmap 级别数
60 4 Bytes of Key Value Data 键值元数据大小

ETC2（Ericsson Texture Compression v2）是 OpenGL ES 3.0 标准格式。每个 4x4 块：

  • RGB 模式：8 字节 / 块（4 bpp）
  • RGBA 模式（EAC）：16 字节 / 块（8 bpp）

本文加载的 KTX 文件包含 ETC2 RGBA8 压缩数据，使用 GL_COMPRESSED_RGBA8_ETC2_EAC（0x9279）内部格式。

ETC2 (KTX) 格式加载

KTX 加载的核心与 PVR 类似：验证标识符 → 解析 64 字节头部 → 处理字节序 → 跳过键值数据段 → 提取第一级 mipmap。不同之处在于 KTX 存储的是压缩数据，需根据 ETC2 的 block size 计算数据量，且需要考虑大/小端字节序。

ASTC 格式详解

ASTC（Adaptive Scalable Texture Compression）是 Khronos 在 2012 年推出的行业领先压缩格式。其核心特点是固定每块 128 位，但支持多种 block size，提供从 8bpp 到 ~0.89bpp 的广泛压缩比。

ASTC 文件格式

由 astcenc 等工具生成的 .astc 文件采用极简的 16 字节头部 + 二进制有效负载：

魔数校验：magic[0] + magic[1] + magic[2] + magic[3] == 443。

图像尺寸解码：decoded_dim = dim[0] + (dim[1] << 8) + (dim[2] << 16)

ASTC Block Size 与 Bits Per Pixel

Block Size 4x4 5x4 5x5 6x5 6x6 8x5 8x6 8x8 10x5 10x6 10x8 10x10 12x10 12x12
Bits Per Pixel 8.00 6.40 5.12 4.27 3.56 3.20 2.67 2.00 2.56 2.13 1.60 1.28 1.07 0.89

ASTC 支持 1-4 通道（R、RG、RGB、RGBA），包括 sRGB 颜色空间。编码端点和权重按块动态选择，128 位按需分配，在相同比特率下质量优于早期格式。

ASTC 的 imageSize 由 block size 决定，每块固定 16 字节（128 位）：

imageSize = ceil(width / block_x) * ceil(height / block_y) * 16

对于 4x4 块：ceil(width / 4) * ceil(height / 4) * 16，每像素恰好 8 位。

更全面的 imageSize 速查表参见 glCompressedTexImage2D 文档。

ASTC 格式加载

ASTC 文件最为精简——16 字节头部后直接跟压缩块数据，无需解析复杂的容器层次。加载的关键是魔数校验和 block size 计算。

PNG 格式详解

PNG（Portable Network Graphics）是使用无损压缩（LZ77 + 过滤器）的图像格式，不是 GPU 原生纹理格式。CPU 解码后才能上传显存，适合 UI 图标、开发调试等场景。

本文使用 stb_image 加载 PNG——这是一个广受欢迎的单头文件图像加载库，由 Sean Barrett 开发。stbi_load() 自动解码 PNG/JPEG/BMP 等格式，返回 RGBA 或 RGB 像素数据。
