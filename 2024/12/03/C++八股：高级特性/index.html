C++八股：高级特性

发表于2024-12-03|更新于2025-12-22|C++C++八股
|总字数:3.6k|阅读时长:11分钟|浏览量:

VERSION：1.0.0

C++11的新特性有哪些？

  1. 自动类型推导 (auto)
  2. 范围for循环 (for (auto x : range))
  3. 右值引用和移动语义 (&&, std::move)
  4. Lambda表达式 ([capture](args) -> ret { body })
  5. 智能指针 (std::unique_ptr, std::shared_ptr, std::weak_ptr)
  6. nullptr
  7. 枚举类 (enum class)
  8. 委托构造函数
  9. override 和 final 关键字
  10. constexpr 函数
  11. 统一的初始化列表 (std::initializer_list 和 {} 语法)
  12. default 和 delete 函数控制
  13. 静态断言 (static_assert)
  14. 元组 (std::tuple)
  15. 新的容器： std::array（固定大小数组），std::forward_list（单链表），std::unordered_map, std::unordered_set（哈希表）。
  16. 正则表达式： std::regex库，原生支持文本模式匹配。
  17. 线程库： std::thread, std::mutex, std::async, std::future等，标准化的多线程支持，不再依赖平台特定API。
  18. 新的算法： std::all_of, std::any_of, std::none_of等。

智能指针的线程安全性如何？多个线程同时操作同一个std::shared_ptr对象是否安全？

  1. 控制块（引用计数）是线程安全的：std::shared_ptr 内部的引用计数增减操作是原子的，保证多个线程同时拷贝/析构指向同一对象的多个 shared_ptr 实例时，引用计数正确无误，不会导致资源多次释放。
  2. 指向的共享对象本身线程不安全：智能指针不保证其管理的T对象的读写安全。你需要额外的同步机制（如互斥锁）来保护*ptr的操作。
  3. 同一个 shared_ptr 实例的写操作非线程安全：多个线程同时读写（如reset、赋值）同一个 shared_ptr 实例（注意，是实例本身，不是副本）是不安全的，需要外部同步。

std::shared_ptr的内部引用计数是如何实现的？它为什么是线程安全的？

  • 实现：shared_ptr 内部包含一个指向堆对象的原始指针和一个指向控制块的指针。控制块中至少包含两个原子操作的引用计数器：use_count（共享所有权计数）和 weak_count（弱引用计数）。
  • 线程安全原因：对引用计数的增减操作（如拷贝构造、赋值、析构）使用原子操作（如 std::atomic::fetch_add 配合 std::memory_order_relaxed 等内存序），确保计数的修改是原子的、不可中断的，从而防止竞态条件导致计数错误。

std::make_shared和直接使用std::shared_ptr构造函数有什么区别？为什么推荐使用std::make_shared？

  • 主要区别：
    1. 内存分配次数：make_shared 会单次分配一块连续内存，同时容纳托管对象和控制块。而 new + shared_ptr 构造函数通常需要两次分配（一次new T，一次分配控制块）。
    2. 异常安全：make_shared 是一个整体操作，能提供更强的异常安全性。例如 f(std::shared_ptr<T>(new T), g())，若g()抛出异常，可能导致new T的内存泄漏。而 f(std::make_shared<T>(), g()) 则避免了此问题。
    3. 内存布局与释放时机：make_shared 对象和控制块内存绑定，即使 use_count 为0，只要 weak_count 不为0（有weak_ptr存在），整个内存块（含对象占用的部分）仍会保留，直到所有weak_ptr也消亡。
  • 推荐原因：更高效（减少一次分配）、更安全（异常安全）、代码更简洁。除非你需要定制删除器或使用分离的原始指针，否则应优先使用make_shared。

智能指针与RAII有什么关系？为什么说智能指针是RAII的典型应用？

  • 关系：智能指针是 RAII（资源获取即初始化） 思想在动态内存管理领域的具体实现。
  • 原因：RAII的核心是将资源的生命周期与对象的生命周期绑定。智能指针（如 unique_ptr, shared_ptr）在构造时获取资源（内存），在析构时自动释放资源（调用delete或自定义删除器）。这种方式确保了即使发生异常、提前返回等，资源也能被正确释放，杜绝了内存泄漏。

介绍一下 C++中的RAII机制？RAII的典型应用有哪些？

  • RAII机制：资源获取即初始化。核心是将必须手动申请/释放的资源（如堆内存、文件句柄、互斥锁、网络连接）的生命周期，托管给一个具有自动生命周期的栈对象。该对象在构造函数中获取资源，在析构函数中释放资源。
  • 典型应用：
    1. 智能指针（unique_ptr, shared_ptr）：管理动态内存。
    2. 锁守卫（std::lock_guard, std::unique_lock）：管理互斥锁，确保退出作用域时解锁。
    3. 容器（std::vector, std::string）：内部管理动态数组内存。
    4. 文件流（std::fstream）：打开文件，析构时自动关闭。

讲一下lambda函数？

  • 概念：Lambda表达式是C++11引入的一种匿名函数对象，用于在调用处就地定义简短的函数逻辑。它简化了代码，特别适合作为回调传递给STL算法。
  • 基本语法：[capture-list] (parameters) -> return-type { body }，其中捕获列表和返回类型（可推导时）可省略。

Lambda表达式的捕获列表有几种方式？值捕获和引用捕获有什么区别？

  • 捕获方式：
    1. 值捕获 [x]：创建时拷贝外部变量x的值，lambda内修改的是副本，不影响外部变量。
    2. 引用捕获 [&x]：捕获外部变量x的引用，lambda内修改直接影响外部变量。
    3. 隐式捕获：[=]（隐式值捕获所有）、[&]（隐式引用捕获所有）。
    4. 混合捕获：[=, &x]（除x引用捕获外，其余值捕获）。
    5. 移动捕获 [x = std::move(y)]（C++14）：将变量y移动进lambda。
  • 核心区别：
    • 值捕获：安全，但可能拷贝开销大；lambda创建时的值快照，后续外部变量变化不影响它。
    • 引用捕获：需注意悬空引用风险（确保lambda执行时被引用的对象仍存活）；能反映外部变量的最新状态。

什么是闭包（Closure）？Lambda表达式和闭包有什么关系？

  • 闭包：是一个函数与其相关的引用环境（即其捕获的外部变量）组合而成的实体。
  • 关系：Lambda表达式是定义闭包的语法糖。当我们编写一个lambda时，编译器会生成一个匿名类（闭包类型）。该lambda的每次实例化都会产生这个类的一个对象，称为闭包对象。捕获列表中的变量会成为该匿名类的成员，从而构成了“闭包”。

std::function是什么？

  • std::function：是一个通用的多态函数包装器。它可以存储、拷贝、调用任何可调用对象，只要其调用签名与 std::function 的模板参数匹配（例如 std::function<void(int)>）。

左值引用和右值引用的区别是什么？

  • 左值引用（T&）：绑定到左值（有名称、有地址、持久存在的对象）。主要用于：
    • 函数参数传递，避免拷贝
    • 创建别名
  • 右值引用（T&&）：绑定到右值（临时对象、字面量、即将销毁的对象）。主要用于：
    • 移动语义：高效转移资源所有权
    • 完美转发：保持值类别转发参数
  • 核心区别：左值引用用于避免拷贝，右值引用用于移动和转发

auto 和 decltype 有何区别？

  • auto：
    • 用于变量声明，根据初始化表达式推导类型
    • 忽略引用&和顶层const（除非显式指定）
    • 示例：auto x = expr; // x的类型是expr值类型
  • decltype：
    • 用于查询表达式类型，不计算表达式
    • 保留引用&和const限定符
    • 示例：decltype(expr) x = expr; // x的类型与expr完全相同

什么是完美转发？其作用及实现方式是什么？

  • 完美转发：函数模板将其参数原封不动（保持值类别、const/volatile属性）转发给其他函数。
  • 作用：
    1. 避免不必要的拷贝
    2. 保持参数的左值/右值特性
    3. 实现通用包装器、工厂函数

实现方式：

1. 通用引用（`T&&`配合引用折叠）
2. std::forward<T> 有条件转换

std::move 与 std::forward 有何区别？

  • std::move：
    • 无条件将参数转为右值引用
    • 实质：static_cast<T&&>(t)
    • 用于启用移动语义，表示资源可被转移
  • std::forward：
    • 有条件转发，保持参数原始值类别
    • 根据模板参数T决定转发为左值还是右值
    • 用于完美转发

使用std::move后原对象会怎样？

原对象处于有效但未指定状态，通常应为空或默认状态，不应再使用除非重新赋值。

移动语义有什么作用，原理是什么？

  • std::move的作用：
    1. 性能优化：避免深拷贝大对象（如vector、string）
    2. 资源转移：实现只能移动不能拷贝的类型（如unique_ptr、thread）
    3. 实现高效容器操作：如vector的resize、push_back等
  • std::move的本质：
    • 强制类型转换：static_cast<T&&>(x)
    • 作用：将左值转换为右值引用，使其可被移动
    • 注意：std::move本身不移动任何东西，只是标记对象为”可移动”
  • std::move的实现机制：
    • 右值引用（T&&）作为触发器，再通过移动构造函数和移动赋值运算符进行资源转移

介绍一下C++中的异常处理机制？

C++异常处理基于try-throw-catch机制，提供结构化的错误处理方式。try块定义监控范围，throw抛出异常对象，catch捕获并处理异常。支持异常类型匹配、栈展开和资源自动清理，通过RAII保证异常安全。

C++异常处理核心组件：

  1. 异常抛出（throw），使用throw表达式抛出异常对象），可以抛出任意类型的对象（内置类型、自定义类、标准异常等），抛出异常会立即终止当前函数执行
  2. 异常捕获（catch），catch块按顺序匹配异常类型，支持异常类型继承层次中的多态捕获，可以使用catch(…)捕获所有异常
  3. 栈展开（Stack Unwinding），从抛出点开始向上查找匹配的catch块，自动调用局部对象的析构函数，保证资源正确释放
  4. 异常规格（Exception Specifications），C++11引入noexcept说明符，替代旧的动态异常规格，提供编译期异常保证

异常处理相比返回错误码有什么优势？

  1. 分离错误处理：正常逻辑和错误处理代码分离，提高可读性
  2. 自动传播：异常自动向上传播，无需每层检查错误码
  3. 类型安全：异常是类型安全的，编译器可以检查类型匹配
  4. 无法忽略：未处理的异常会导致程序终止，错误不会被静默忽略
  5. 构造函数支持：构造函数无法返回错误码，只能使用异常报告失败

什么是栈展开？在异常处理中起什么作用？

栈展开是从异常抛出点开始，沿着调用链向上查找匹配catch块的过程。在这个过程中：

  • 自动调用局部对象的析构函数
  • 释放自动存储期的资源
  • 维护调用栈的完整性
  • 保证资源不泄漏（结合RAII） 栈展开确保了即使发生异常，程序也能正确地清理资源。

为什么应该按引用捕获异常？

按引用捕获异常有多个优势：

  • 避免切片：捕获派生类异常时保持多态性
  • 避免拷贝：不需要复制异常对象，提高性能
  • 保持原始异常：能够重新抛出原始异常对象
  • 一致性：与标准库异常处理方式一致

在析构函数中抛出异常会怎么样？

在析构函数中抛出异常是危险的，因为：

  • 双重异常：如果栈展开过程中析构函数抛出异常，会立即调用terminate
  • 资源泄漏：可能中断其他资源的清理过程
  • 未定义行为：C++标准规定这种情况会导致程序终止
  • 所以析构函数应该用noexcept声明，并吞掉所有异常

在构造函数中抛出异常会导致什么问题？如何正确处理？

问题：

  1. 如果构造函数中抛出异常，那么对象的析构函数将不会被调用（因为对象构造不完整，C++不会调用析构函数）。
  2. 但是，在构造函数中，如果成员变量是带有析构函数的类类型，那么这些成员变量（它们已经被构造）的析构函数会被调用。
  3. 如果在构造函数中分配了动态内存或打开了文件等资源，并且构造函数在之后抛出异常，那么这些资源需要被清理，否则会导致资源泄漏。

避免措施：

  1. 使用智能指针管理动态内存：如果构造函数中分配内存，可以使用智能指针（如std::unique_ptr）来管理。这样，当构造函数抛出异常时，已经构造的成员变量（包括智能指针）会被销毁，智能指针的析构函数会自动释放内存。
  2. 使用RAII类管理资源：对于其他资源（如文件句柄、锁等），也应该使用RAII类来管理。这样，当构造函数抛出异常时，这些RAII类的析构函数会负责释放资源。
  3. 避免在构造函数中做可能抛出异常的操作，如果不可避免，则确保在异常抛出前已经分配的资源能够被正确释放。
