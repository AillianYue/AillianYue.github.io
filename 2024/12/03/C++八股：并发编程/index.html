C++八股：并发编程

进程线程了解吗？有什么区别？

  • 进程是操作系统资源分配的基本单位，每个进程有独立的代码、数据空间和系统资源（如内存、文件句柄）。进程间通信（IPC）成本高，如管道、消息队列、共享内存。
  • 线程是CPU调度和执行的基本单位，是进程内的一条执行路径。同一进程下的多个线程共享进程的资源（如全局变量、堆空间），这使得线程间通信非常高效，但同时也带来了同步问题（如竞态条件）。

并发和并行了解么？有什么区别？

  • 并发（Concurrency）是指多个任务在同一时间段内交替执行，在单核或多核系统中都可以发生。它关注的是任务的结构，使得多个任务可以同时被处理（但未必同时执行）。
  • 并行（Parallelism）是指多个任务在同一时刻同时执行，这要求多核或多处理器系统。
  • 区别：并发是逻辑上的同时，通过时间片轮转实现；并行是物理上的同时。并发关注的是任务拆解和响应能力，并行关注的是任务执行效率。

C++中如何创建和管理一个线程？

  • 使用std::thread类来创建线程，构造函数中传入可调用对象（函数、函数指针、lambda表达式等）和参数。
  • 管理线程包括：等待线程结束（join）、分离线程（detach）、获取线程ID、线程休眠（std::this_thread::sleep_for）等。

std::thread 的 join() 和 detach() 有什么区别？

  • join()：阻塞当前线程，直到被调用的线程执行完毕。它还会清理线程相关的存储部分，这样std::thread对象就不再与已结束的线程相关联。之后，该std::thread对象变为不可连接（joinable() == false）。
  • detach()：将当前线程与创建的线程分离，使得它们彼此独立执行。分离后，std::thread对象不再代表任何线程，变为不可连接。分离线程的资源会在线程结束时自动回收。
  • 注意：在销毁std::thread对象之前，必须确保已经调用join()或detach()，否则程序会调用std::terminate()终止。

如果一个线程没有被 join() 也没有被 detach()，会发生什么？

会发生程序终止（调用std::terminate()）。

原因：

  • std::thread的析构函数会检查线程是否可连接（joinable）
  • 如果可连接，析构函数调用std::terminate()终止整个程序

所以，std::thread 对象在销毁前必须调用 join() 或 detach()。

什么是线程局部存储（Thread Local Storage）？

  • 线程局部存储是一种机制，允许每个线程拥有该变量的独立副本，每个线程对该变量的读写都不会影响其他线程。在C++中，使用thread_local关键字声明变量。线程局部存储的变量在第一次使用时初始化，并且在线程结束时销毁。

什么是原子操作？为什么需要原子操作？

原子操作：不可分割的操作，要么完全执行，要么完全不执行，不会出现中间状态。

为什么需要：

  1. 避免数据竞争：多个线程同时读写同一内存位置
  2. 无需锁的同步：比互斥锁更轻量
  3. 保证操作完整性：如计数器自增i++在CPU层面不是原子的

如何理解C++中的 atomic ？

  • std::atomic是C++标准库提供的模板类，用于定义原子类型。它提供了一系列成员函数和操作符重载，使得对共享数据的操作是原子的。它保证了在多线程环境下，对该对象的读写操作是原子的，不会出现数据竞争。

C++中的 std::atomic 提供了哪些基本操作？它如何保证原子性？

  • 基本操作包括：
    • 加载（load）：读取原子变量的值。
    • 存储（store）：写入原子变量的值。
    • 交换（exchange）：交换内容，返回之前的值。
    • 等
  • 保证原子性的方式：通常通过底层硬件支持的原子指令（如x86的LOCK前缀指令）来实现，避免在操作期间被中断。对于不支持原子指令的类型，std::atomic可能会使用互斥锁来模拟原子性，但这样性能会下降。

C++11 中多线程编程与锁你了解吗？

C++11 引入了多线程支持，包括线程、互斥量、条件变量、原子操作等

  1. 线程（thread）。C++11 提供了 std::thread 类来创建和管理线程。需要包含 头文件。
  2. 互斥量（mutex）C++11 提供了多种互斥量类，用于保护共享数据。最基本的互斥量，提供 lock()、unlock() 和 try_lock() 方法。需要包含 <mutex> 头文件。
  3. RAII锁管理器
    1. lock_guard。为了避免忘记解锁，C++11 提供了 std::lock_guard，在构造时锁定互斥量，析构时自动解锁。
    2. unique_lock。std::unique_lock 比 std::lock_guard 更灵活，可以延迟锁定、手动解锁等。
  4. 条件变量（condition_variable）。条件变量用于线程间的同步，允许线程阻塞直到某些条件成立。需要包含 <condition_variable> 头文件。
  5. 原子操作（atomic）。C++11 提供了原子类型和操作，用于无锁编程。需要包含 <atomic> 头文件。

什么是竞态条件（Race Condition）和数据竞争（Data Race）？它们有什么区别？

  • 竞态条件：多个线程的执行顺序不确定，导致程序结果依赖于执行时序。
  • 数据竞争：多个线程未同步地访问同一内存位置，且至少有一个是写入操作。
  • 区别：所有数据竞争都是竞态条件，但并非所有竞态条件都是数据竞争。

请解释互斥锁（Mutex）的基本工作原理，以及为什么需要互斥锁？

工作原理：互斥锁是二进制信号量，只有”锁定”和”解锁”两种状态：

  1. 线程尝试获取锁（lock）
  2. 如果锁空闲，线程获得锁并继续执行
  3. 如果锁被占用，线程阻塞等待
  4. 持有锁的线程解锁后，唤醒等待线程

为什么需要：

  1. 保护临界区：确保同一时刻只有一个线程访问共享资源
  2. 保证操作原子性：复杂操作不会被其他线程中断

lock_guard和unique_lock有什么区别？ 什么时候应该用哪一个？

  • lock_guard 更轻量，构造时立即锁定，析构时自动解锁，不支持手动控制
  • unique_lock 更灵活，支持延迟锁定、手动锁定解锁、条件变量配合使用性能上lock_guard更好，unique_lock功能更全但稍重

使用：

  • 简单作用域锁：使用lock_guard，自动加锁解锁
  • 需要灵活性：使用unique_lock，手动加锁解锁
  • 配合条件变量：必须用unique_lock

什么是条件变量（Condition Variable）？它解决什么问题？

条件变量：线程间通知机制，允许线程等待特定条件成立。

解决的问题：

  1. 忙等待（Busy-waiting）：避免线程循环检查条件，浪费CPU
  2. 线程协调：实现生产者-消费者、读者-写者等模式
  3. 事件驱动：等待异步操作完成

请解释条件变量的“虚假唤醒”（Spurious Wakeup）是什么，以及如何正确处理？

虚假唤醒：等待的线程在没有收到通知的情况下被唤醒。

原因：

  1. 底层实现限制（如POSIX条件变量允许虚假唤醒）
  2. 硬件/操作系统中断
  3. 编译器优化

正确处理：使用while循环

条件变量为什么要和unique_lock配合使用？

因为 condition_variable::wait() 需要能够临时释放锁让其他线程运行，并在条件满足时重新获取锁。unique_lock支持这种灵活的锁定机制，而lock_guard在作用域内始终保持锁定状态。

什么是死锁？什么情况下会出现死锁？如何避免？

死锁是多个线程互相等待对方释放资源导致的永久阻塞。

死锁产生的四个必要条件（Coffman条件）：

  1. 互斥条件：一个资源每次只能被一个进程使用。
  2. 请求与保持条件：一个进程因请求资源而阻塞时，对已获得的资源保持不放。
  3. 不剥夺条件：进程已获得的资源，在未使用完之前，不能强行剥夺。
  4. 循环等待条件：若干进程之间形成一种头尾相接的循环等待资源关系。

避免方法：

  1. 统一锁定顺序
  2. 使用std::lock同时锁定多个mutex
  3. 使用RAII包装器管理锁
  4. 避免在持有锁时调用未知代码
  5. 使用超时机制

讲一下C++中的互斥锁和自旋锁

  • 自旋锁（Spinlock）通过忙等待实现同步，在用户态循环检查锁状态，适用于锁持有时间短的场景。
  • 互斥锁（Mutex）通过系统调用实现阻塞，线程会进入睡眠状态，适用于锁持有时间较长的场景。

自旋锁避免上下文切换开销但浪费CPU周期，互斥锁节省CPU资源但引入切换开销。

自旋锁特性：

  • 实现机制：在用户空间忙等待，不断循环检测锁状态
  • CPU消耗：高，即使等待也在持续消耗CPU周期
  • 上下文切换：无，线程始终保持运行状态
  • 响应延迟：低，锁释放后能立即获取
  • 实现复杂度：相对简单，通常基于原子操作

互斥锁特性：

  • 实现机制：通过操作系统内核调度，线程阻塞并进入睡眠
  • CPU消耗：低，等待时不消耗CPU资源
  • 上下文切换：有，涉及用户态到内核态切换
  • 响应延迟：高，需要唤醒睡眠线程
  • 实现复杂度：相对复杂，依赖操作系统支持

性能关键因素：

  • 锁竞争强度：高竞争下自旋锁性能急剧下降
  • 临界区大小：短临界区适合自旋锁，长临界区适合互斥锁
  • CPU核心数：多核系统更适合自旋锁
  • 调度策略：实时系统可能偏好自旋锁

一个项目为什么要用到多线程？

  1. 提高CPU利用率：通过利用多核处理器，将任务分解到多个线程中并行执行，可以显著减少计算密集型任务的执行时间。
  2. 避免阻塞：在图形用户界面（GUI）应用程序中，使用多线程可以避免长时间的计算任务阻塞事件循环，从而保持界面的响应性。
  3. 提高吞吐量：多线程允许程序在等待一个任务（如I/O操作）完成的同时，执行其他任务，从而更有效地利用系统资源。

C++11 中智能指针的线程安全性如何？

智能指针的引用计数是线程安全的，但是智能指针对象本身（作为对象实例）的修改需要同步，除非使用原子操作。智能指针所管理的对象的线程安全性由用户负责。

  • shared_ptr：引用计数是线程安全的，指向的对象本身并不是线程安全的
  • unique_ptr：移动语义和所有权转移非线程安全
  • weak_ptr：与shared_ptr类似

shared_ptr的引用计数是如何保证线程安全的？

通过原子操作实现。控制块中的引用计数使用std::atomic类型，所有的增减操作都是原子的，使用适当的内存序（通常是memory_order_relaxed）来保证多线程环境下的正确性。

为什么对同一个shared_ptr实例的写操作需要同步？

因为shared_ptr的写操作（如reset、赋值）涉及多个步骤：修改内部指针、调整引用计数等。如果没有同步，多个线程同时修改会导致数据竞争和未定义行为。

如何安全地在多线程环境中传递shared_ptr？

有几种安全方式：

  1. 值传递：函数参数按值传递，自动增加引用计数
  2. const引用 + 内部复制：避免不必要的引用计数操作
  3. 使用atomic_shared_ptr（C++20）：提供原子操作接口

为什么要有线程池，线程太多会怎样？

为什么需要线程池：

  1. 降低开销：线程创建/销毁代价高（内核资源分配）
  2. 控制并发度：避免系统资源耗尽
  3. 任务管理：统一调度，提高响应速度
  4. 资源复用：重用已创建的线程

线程太多的危害：

  1. 内存消耗：每个线程需要栈空间（默认1-8MB）
  2. 上下文切换开销：大量线程导致CPU时间浪费在切换上
  3. 缓存失效：频繁线程切换导致缓存不命中
  4. 系统限制：操作系统有最大线程数限制
  5. 调度延迟：太多线程导致调度延迟增加

经验值：最佳线程数 ≈ CPU核心数 × (1 + 等待时间/计算时间)

请描述线程池（Thread Pool）的工作原理，并说明其优势。

工作原理：

  1. 初始化：创建固定数量的工作线程
  2. 任务队列：提交的任务放入队列
  3. 工作循环：线程从队列获取任务执行
  4. 优雅关闭：处理完所有任务后关闭

优势：

  1. 性能：避免频繁创建/销毁线程
  2. 可控性：限制并发线程数量
  3. 可管理性：统一异常处理、资源管理
  4. 响应性：任务排队，快速响应新请求
  5. 可扩展性：支持任务优先级、负载均衡

如何实现一个简单的线程池？请描述关键设计点。

关键设计点：

  1. 任务表示：使用std::function<void()>封装任意任务
  2. 线程管理：固定数量线程，避免动态创建
  3. 任务调度：FIFO队列，公平调度
  4. 优雅关闭：通过标志位和空任务通知线程退出
  5. 异常安全：确保线程异常不影响线程池
  6. 结果获取：可扩展支持std::future返回结果
